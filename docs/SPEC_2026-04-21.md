# ARIA — IA Audit Tool · Technical Specification

## 1. Purpose

Web platform to run structured AI-audit engagements: capture process context, score AI-sovereignty, map activities, identify use cases, and follow them through POC and production implementation.

## 2. Stack

## 3. Domain model

Collections (MongoDB, db `aria-audit`):

- `users` — authentication, roles `admin | consultant | viewer`.
- `audits` — top-level engagement: client, sector, classification, status, dates.
- `processes` — business processes scoped to an audit (`auditId`, `procId`, department, maturity, priority).
- `usecases` — AI use cases derived from a process; scoring, type (`AI_TYPE_LABELS`), time-saved entries.
- `pocs` — proofs-of-concept spawned from use cases.
- `roadmaps` — implementation roadmap per audit.
- `counters` — monotonic counters (e.g. procId sequence).
- `articles` — knowledge-base entries (reserved).

Core types are declared in [lib/types.ts](lib/types.ts); Mongoose schemas in [lib/models/](lib/models/).

## 4. Audit workflow — the five blocks

Each process inside an audit is audited via sequential blocks. UI: [app/(app)/audits/[auditId]/processes/[procId]/](<app/(app)/audits/[auditId]/processes/[procId]/>).

| Block | Name            | Responsibility                                                             |
|-------|-----------------|----------------------------------------------------------------------------|
| B1    | Context         | Formal name, department, stakeholders, profiles, attitudes toward AI.      |
| B2    | Sovereignty     | Axis-by-axis scoring → Sovereignty Index (`lib/calculations.ts`).           |
| B3    | Process Map     | Activities, tasks, profile hours, attachments; computes time/cost.         |
| B5    | Use Cases       | Opportunity identification, per-UC scoring, generation from B3.             |

> Note: "B4" no longer exists as a label; all legacy "B4" strings were renamed to "B5" in commit `254e3395`. Route segment `b5` is the canonical one for use cases.

## 6. Cross-cutting behaviour

- **Role gating**: `viewer` read-only; `consultant` full audit CRUD on assigned audits; `admin` global + user management. Enforced by [middleware.ts](middleware.ts) and per-route `requireRole`.
- **Rate limiting**: login endpoint rate-limited (IP-based).
- **Zod validation**: every POST/PATCH body is validated before DB I/O.
- **Breadcrumbs**: two nested layouts — [app/(app)/audits/[auditId]/layout.tsx](<app/(app)/audits/[auditId]/layout.tsx>) handles audit sub-sections, [app/(app)/audits/[auditId]/processes/[procId]/layout.tsx](<app/(app)/audits/[auditId]/processes/[procId]/layout.tsx>) handles process and block-level pages.
- **Unsaved-change guards**: `useBeforeUnload` hook wired on B1, B2, B3 and B5 editors.
- **Save indicator**: SaveIndicator component surfaces dirty/saving/saved states.
- **Confirm modals**: destructive actions (archive, delete) use `ConfirmModal`, no `window.confirm`.
- **Progress**: `BlockProgressBar` computes per-process completion across B1/B2/B3/B5.

## 7. Data persistence

- Local dev: MongoDB in Docker container `aria-mongo` (image `mongo:7`), port `27017`, data volume anonymous (see deployment notes).
- App container reaches DB via `host.docker.internal:27017` ([docker-compose.yml](docker-compose.yml)).
- Production: `MONGODB_URI` environment variable, recommended managed service (Atlas) with auth + backups.

Backup / restore contract:

```bash
docker exec aria-mongo mongodump --db aria-audit --archive=/tmp/aria-audit.archive --gzip
docker cp aria-mongo:/tmp/aria-audit.archive ./backups/aria-audit-YYYYMMDD.archive.gz
mongorestore --uri=$MONGODB_URI --archive=<file>.archive.gz --gzip
```

A `migrate_mongo_to_postgres.py` script exists for optional migration to PostgreSQL.
